# PRD: 계정탈취 부정결제 방지 — 간편결제 FDS (MVP)

> 작성자: 이다영 (4조) / 작성일: 2026-07-16 / 브랜치: `feat/prd-dayoung`
> 표기 규칙: **[확인]** = 출처·분석으로 검증된 값, **[가정]** = 아직 검증 안 된 전제

## TL;DR

개인정보 유출로 크리덴셜이 도용된 **이커머스 간편결제 이용자**가, 낯선 기기·위치에서 시도되는 **상품권 등 환금성 부정결제가 승인되기 전에** 단계별 인증으로 걸러져 **자산과 계정 통제권을 지키도록** 한다. 정상 이용자는 평소처럼 결제한다.

---

## Background

- **대규모 크리덴셜 유출이 실제로 발생했다.** 2026년 5월 티빙에서 1,953만 명의 아이디·비밀번호(단방향 암호화)·CI·DI·환불계좌번호가 유출됐다. 국내 개인정보 유출 사고 중 네 번째 규모다. **[확인]** (출처: [미디어오늘](https://www.mediatoday.co.kr/news/articleView.html?idxno=335252), [이투데이](https://www.etoday.co.kr/news/view/2596110))
- **유출은 한 서비스에서 끝나지 않는다.** CJ ONE(가입자 약 3,100만)은 크리덴셜 스터핑 2차 피해를 막기 위해 연동 계정을 2026년 6월 10일부터 7월 2일까지 **전면 잠금**했다. **[확인]** (출처: [보안뉴스](https://m.boannews.com/html/detail.html?idx=144078))
- **기존 대안의 한계 — 전면 잠금은 미탐 0이지만 오탐이 폭발한다.** 자체 분석에서 "로그인 성공 + 낯선 국가 + 낯선 기기" 룰은 실제 계정탈취 11건을 **전부** 잡았지만(Recall 1.00), 경보가 13만 건 발생해 정밀도 0.00008이었다. 즉 정상 이용자를 대량으로 막는 방식이다. **[확인]** (RBA 로그인 데이터 50만 건 분석)
- **지금 해결해야 하는 이유:** 유출된 크리덴셜은 다른 서비스에 대입되는데, 이커머스 간편결제 계정에는 카드·계좌가 이미 등록돼 있어 계정이 뚫리는 순간 결제수단이 통째로 노출된다. 잠금이 아닌 **선별 탐지**가 필요하다.

## Problem Statement

- **Current Situation:** 이용자는 이커머스 계정에 카드·계좌를 등록해두고 간편결제를 쓴다. 여러 사이트에 같은 비밀번호를 재사용하는 경우가 많다.
- **Pain Point:** 계정이 도용돼도 **본인은 결제가 승인된 뒤에야 안다.** 공격자가 이메일·비밀번호를 먼저 바꾸면 알림조차 받지 못한다. 상품권·기프트카드는 승인 즉시 사용·전송돼 회수가 불가능하다.
- **Impact:** 고객은 결제 자산과 계정 통제권을 동시에 잃고, 회사는 차지백·보상 비용과 브랜드 신뢰를 잃는다. 대규모 발생 시 "○○페이 뚫렸다" 보도로 이어진다.

**정량 근거**

| 지표 | 값 | 구분 |
| :--- | :--- | :--- |
| 카드 미승인(CNP) 부정결제 비율 | 3.5% (IEEE-CIS 59만 건 기준) | **[확인]** |
| 주소 국가 불일치 결제의 사기율 | 12% (전체 3.5% 대비 3.4배) | **[확인]** |
| 계정탈취 로그인 중 낯선 국가·기기 비율 | 100% (정상은 각 36%·41%) | **[확인]** |
| 사기 라벨 확정까지 소요 기간 | 차지백 2~8주 / 직접 신고 1~7일 | **[가정]** (업계 통상, 실데이터 검증 필요) |

## Target Persona

**김지현 (28세, 직장인, 서울 거주)**

- 네이버페이·쿠팡페이에 카드 3장을 등록해두고 주 2~3회 모바일로 결제한다. 결제는 대부분 5만 원 이하.
- 가입한 서비스가 20개가 넘고, 그중 상당수에 **같은 비밀번호를 재사용**한다.
- 티빙 유출 안내 문자를 받았지만 "어느 사이트에 같은 비밀번호를 썼는지" 기억나지 않아 그냥 넘겼다.

| 구분 | 내용 |
| :--- | :--- |
| Pains | 내 계정이 이미 뚫렸는지 알 방법이 없다. 결제 알림을 놓치면 그대로 피해가 확정된다. |
| Needs | 내가 아닐 때만 막아주고, **나일 때는 방해받지 않기**. 막혔다면 왜 막혔는지 알고 싶다. |
| 첫 고객 선정 이유 | 결제수단이 이미 등록돼 있고(피해 즉시성) + 비밀번호를 재사용하며(공격 노출) + 모수가 가장 크다. |

## Goals

| 관점 | 목표 |
| :--- | :--- |
| User Goal | 도용된 결제는 승인 전에 멈추고, 본인 결제는 마찰 없이 통과한다. 막혔을 때 이유와 해제 방법을 안다. |
| Product Goal | "무조건 차단하는 FDS"가 아니라 **"한 번 확인하는 FDS"**. 미탐을 줄이면서 오탐 비용을 인증 단계로 흡수한다. |
| Business Goal | 부정결제 피해액과 차지백 비용을 줄이고, 유출 사고 국면에서도 결제 신뢰를 유지한다. |

## Success Metrics

**북극성 지표**

- **월간 부정결제 확정 피해 금액(원)** — 차지백·신고로 사기 확정된 결제의 피해액 합계. 이벤트 후 **30일 확정 기준**으로 집계(라벨 지연 반영).

**보조 지표**

- 승인 전 차단율: 확정 사기 중 승인 전에 막힌 비율
- Recall(미탐 최소화가 1순위), Precision@K(검토 인력 기준)

**가드레일 지표 (넘으면 정책 롤백)**

| 지표 | 한계선 |
| :--- | :--- |
| 정상 결제 중 추가 인증·보류·차단을 겪은 비율 | **≤ 2.0%** |
| 운영팀 일일 경보량 | ≤ 500건 |
| P95 조치 시간 | ≤ 30분 |

> 불균형 데이터이므로 **Accuracy는 지표로 쓰지 않는다.** 전부 정상이라고 예측해도 96.5%가 나온다.

## Key Scenario

1. **Context** — 티빙 유출 3주 후, 공격자가 확보한 아이디·비밀번호 조합을 여러 이커머스에 대입한다.
2. **Problem Moment** — 김지현의 계정에 로그인 성공. 공격자가 이메일을 바꿔 알림을 차단하고, 등록된 카드로 200만 원 상품권을 결제하려 한다.
3. **Product Interaction** — FDS가 이 세션을 **낯선 기기 + 낯선 국가 + 최근 이메일 변경 + 고환금성 카테고리**로 판정해 위험 점수를 매기고, 결제를 **보류**한 뒤 **변경 전 원래 연락처로** 본인 확인을 요청한다.
4. **Resolution** — 김지현은 자신이 하지 않은 결제 알림을 받고 "내가 아님"을 선택한다. 결제가 차단되고 계정이 보호 잠금된다.
5. **Outcome** — 200만 원이 빠져나가지 않았고, 김지현은 비밀번호를 재설정해 계정을 되찾는다. 회사는 차지백 비용을 치르지 않는다.

## Solution & Key Features

**기존 대안의 한계**

- 전면 계정 잠금(CJ ONE 방식): 미탐은 0이지만 정상 이용자까지 차단 → 오탐 비용이 과도하다. **[확인]**
- 단일 룰(신규 기기만, 고액만): 여행·기기 교체 같은 정상 예외를 구분하지 못한다.
- 사후 차지백 대응: 상품권은 승인 즉시 현금화되어 회수가 불가능하다.

**MVP 핵심 기능 (4개)**

| # | 기능 | 연결된 문제 |
| :--- | :--- | :--- |
| 1 | **세션 리스크 스코어링** — 로그인 시점에 신규 기기/신규 국가/동일 IP 다계정 접속/로그인 실패 횟수를 집계해 세션 위험 점수를 부여 | 계정 도용을 결제 전에 감지 |
| 2 | **결제 시점 위험 판정** — 세션 위험 점수 + 결제 신호(고환금성 카테고리, 평소 대비 금액 이탈, 발신·수신 이메일 불일치)를 결합해 최종 점수 산출 | 승인 순간에 막아야 회수 가능 |
| 3 | **위험 수준별 단계 조치** — 자동 승인 → 추가 인증(OTP) → 결제 보류 + 원래 연락처 알림 → 차단 + 계정 보호 잠금 | 차단 일변도의 오탐 비용을 인증으로 흡수 |
| 4 | **운영 검토 큐** — 위험 점수 상위 K건을 검토 사유(왜 걸렸는지)와 함께 제공 | 제한된 인력으로 판단 가능하게 |

> Should: 차지백·신고 결과를 라벨로 수집해 재학습하는 파이프라인 (라벨이 쌓인 뒤 지도학습으로 전환)

## UX Principles

- **핵심 감정:** "막혔다"가 아니라 **"확인받았다"**. 기본값은 차단이 아니라 확인 한 번.
- **정보 우선순위:** ① 지금 무슨 일이 일어났나 → ② 내가 뭘 하면 되나 → ③ 왜 그렇게 판단했나.
- **금지·주의 표현:** "해킹당했습니다" 같은 불안 조장 표현 금지. 확정되지 않은 사실을 단정하지 않는다. 탐지 규칙의 구체적 임계값은 노출하지 않는다(우회 방지).
- **정상 예외 배려:** 해외여행, 기기 교체, 명절 상품권 대량 구매는 **정상일 수 있다**는 전제로 문구를 쓴다.
- **시각화 원칙:** 운영 검토 화면은 복잡한 차트보다 **위험 점수 + 걸린 이유 3줄 + 권장 조치**를 크게.

## Constraints & Assumptions

**Constraints (고정된 한계)**

- **라벨이 늦게 도착한다.** 차지백 확정까지 2~8주, 피해자가 인지 못 하면 사기가 정상으로 남는다(라벨 누락).
- **극단적 불균형.** 양성 비율 약 3.5% 수준 → 학습 시에만 보정하고 평가는 실제 비율로 한다.
- **개인정보 제약.** 기기 식별값·IP는 개인정보에 해당할 수 있어 처리방침 명시 필요. 결제·신용정보는 신용정보법이 별도 규율.
- **실시간 요구.** 결제 승인 순간에 판정해야 하므로 무거운 집계는 사전 계산(리스크 컨텍스트 테이블) 필요.

**Assumptions (검증 필요) + 검증 방법**

| 가정 | 검증 방법 |
| :--- | :--- |
| 상품권·기프트카드가 ATO 부정결제의 주요 현금화 경로다 | 확정 사기 건의 상품 카테고리 분포 분석 |
| 오탐이 2% 이하면 이탈에 유의미한 영향이 없다 | 인증 노출군/비노출군 A/B로 결제 포기율·재방문율 비교 |
| 이메일 변경 직후 결제는 ATO의 강한 신호다 | 계정 이벤트 로그와 사기 라벨 조인해 리프트 측정 |
| 운영팀이 하루 500건까지 검토 가능하다 | 실제 검토 소요시간 측정(건당 처리시간 × 인력) |

## Open Issues

- **데이터:** 로그인 시스템과 결제 시스템을 잇는 결합 파이프라인 구축 주체와 SLA는? 결합 실패 시 결측 처리 정책은?
- **법무 / 개인정보:** 동일 IP의 타 계정 정보를 참조하는 관계 분석이 **수집 목적 외 이용**에 해당하는가? 처리방침에 어떻게 명시할 것인가?
- **UX:** 결제 보류 시 대기 허용 시간은 몇 분인가? 그 시간을 넘기면 자동 승인인가 자동 취소인가?
- **기술:** 평가 창을 30일로 잡는 것이 적정한가? 차지백 8주 건은 어떻게 반영할 것인가?
- **비즈니스:** 오탐으로 인한 결제 포기 손실과 미탐으로 인한 피해액 중 어느 선에서 임계값을 잡을 것인가?
