# PRD - 이다영

> **한 줄 정의:** 탈취된 카카오 계정에서 시도되는 **기프티콘 결제**를 승인 전에 차단하고 **연결된 서비스까지 함께 보호**하는 간편결제 FDS. **유출 노출 정황이 있는 계정일수록 더 엄격하게** 판정한다.
> 
> 
> **대상 서비스:** 소셜 로그인(SSO)과 자체 간편결제를 함께 제공하는 플랫폼 - **카카오**(카카오톡·카카오페이·선물하기) 기준
> 

---

## 1. 문제 (Problem)

### 1-1. 이미 벌어진 일이다 (실제 사례)

| 사례 | 무슨 일이 있었나 |
| --- | --- |
| **스타벅스 코리아 (2023.7)** | 외부에서 확보한 아이디·비밀번호를 대입하는 **크리덴셜 스터핑**으로 **해외 IP**를 통해 앱에 부정 로그인. 고객 **90여 명**의 **충전금 약 800만 원**이 결제 도용됨. 회사는 IP 차단 후 **충전금 전액 보전**. |
| **티빙 (2026.5)** | **1,953만 명**의 아이디·비밀번호·CI/DI·환불계좌 유출. 국내 유출 사고 중 4번째 규모. |
| **CJ ONE (2026.6~7)** | 티빙 유출 여파로 **연동 계정을 선제적으로 잠금**(6.10~7.2). 2차 피해 확산을 막기 위한 조치. |
| **23andMe (2023, 해외)** | 크리덴셜 스터핑으로 **690만 명** 정보 유출. 원인 중 하나가 **“하나의 계정이 침해되면 연결된 수백~수천 명까지 노출되는 연결 구조”**. |

**보편성:** 개인정보보호위원회·KISA 「2024년 개인정보 유출 신고 동향」에서 크리덴셜 스터핑은 **해킹 유형 중 3번째로 많은 공격 기법**이며, Verizon 2024 DBIR은 도용된 인증정보 기반 공격이 **전체 침해의 약 1/3**을 차지한다고 보고했다.

> **핵심:** 스타벅스 사례는 우리 시나리오와 구조가 같다. **해외 IP + 크리덴셜 스터핑 + 선불 자산(충전금) 도용 + 회사가 비용 부담.** 게다가 대응이 전부 **사후**였다.
> 

### 1-2. 유출된 정보가 어떻게 결제로 이어지나

**비밀번호는 암호화돼 있는데 어떻게 뚫리나?**

- 유출된 비밀번호는 보통 **단방향 암호화(해시)** 되어 그 자체로는 바로 쓸 수 없다. 티빙 유출도 마찬가지였다.
- 그런데 공격자는 **해시를 깰 필요가 없다.** 과거 수많은 유출에서 **이메일과 평문 비밀번호가 짝지어진 목록(콤보 리스트)** 이 이미 유통되고 있기 때문이다.
- 즉 공격은 **“비밀번호를 알아내는” 것이 아니라 “이미 아는 조합을 밀어넣는” 것**이다. 그래서 이름이 크리덴셜 **스터핑**(밀어넣기)이며, **비밀번호를 재사용한 계정**이 뚫린다.

**유출 항목별 역할 (CI를 정확히 이해하기)**

| 유출 항목 | 공격에서의 역할 |
| --- | --- |
| 이메일·아이디 | **표적 특정** — 누구를 공격할지 고른다 |
| 비밀번호(재사용 시) | **실제 침입 경로** ← 진짜 동력 |
| **CI/DI** | **확산 지도** — 서비스가 달라도 동일인 매칭이 가능해, 어느 서비스에 걸쳐 있는지 파악 |
| 환불계좌·연락처 | 피싱·사회공학 재료 |

> **CI는 침입 수단이 아니라 확산 지도다.** CI로 로그인할 수는 없지만, “이 사람이 카카오에도 있다”를 알려준다. CJ ONE이 연동 계정을 잠근 근거도 이 연결 고리다.
> 

### 1-3. 유출 출처에 따라 위협이 달라진다

| 구분 | 공격자가 얻는 것 | 위협의 성격 |
| --- | --- | --- |
| **A. 플랫폼 본체 유출** 
(카카오 자체) | 아이디 + 비밀번호(해시) + 계정 구조 | **연결된 모든 서비스가 동시 노출.** 단일 장애점이 터진 상황 |
| **B. 제3자 서비스 유출** 
(티빙 등) | 이메일·이름·CI (카카오 비밀번호는 안전) | CI로 **표적을 특정**해 카카오에 **밖에서 스터핑 시도**. 외부 유입 공격 |

### 1-4. SSO에서 피해가 연쇄된다

- 카카오 계정은 **단일 장애점**이다. 계정 하나가 뚫리면 그 계정으로 로그인한 **연결 서비스 전체가 동시에 노출**된다.
- 실제로 카카오는 `카카오계정 > 계정 이용 > 연결된 서비스 관리`에서 연결 서비스 목록과 해제 기능을 제공한다. **플랫폼은 어떤 서비스가 연결돼 있는지 이미 알고 있다.**
- 계정 장악 후 도달 경로는 **세 갈래**다.

| 경로 | 설명 | 위험도 |
| --- | --- | --- |
| **① 앱 자체** | 카카오톡·카카오페이. 
**결제수단이 등록돼 있고 선물하기로 기프티콘 즉시 구매 가능** | 최고 |
| **② 기존 연결 서비스** | 이미 로그인 전적이 있어 **비밀번호 없이 줄줄이 통과** | 높음 |
| **③ 신규 가입** | 공격자가 새로 연결 | 낮음 |

**왜 기프티콘(환금성 상품)에 집중하는가**

우리 제품의 가치는 “승인 전에 막는 것”인데, **그 가치는 상품마다 다르다.**

| 상품 유형 | 사후 대응 가능성 | 실시간 차단의 가치 |
| --- | --- | --- |
| 실물 배송 상품 | 배송 중단·회수 가능 | 상대적으로 낮음 |
| **기프티콘·충전금** | **승인 즉시 전송·사용 → 회수 수단 없음** | **가장 높음** |
- **회수가 불가능한 영역이 곧 실시간 차단이 유일한 방어인 영역이다.** 스타벅스가 충전금 800만 원을 전액 보전한 것도 회수가 불가능해 보전 외에 방법이 없었기 때문이다.
- **오탐 비용도 줄어든다.** 모든 결제에 엄격한 룰을 걸면 불편이 전면적으로 커지지만, 환금성 카테고리에 한정하면 영향 범위가 좁다.
- **공격자가 실제로 선호한다.** 실물은 배송지가 필요해 추적되지만 기프티콘은 익명 전송·재판매가 가능하다.

> 단, 카테고리는 **단독 조건이 아니라 위험 가중치**로 사용한다. 이 영역만 막으면 공격이 다른 카테고리로 이동하기 때문이다(부록 B 참조).
> 

### 1-5. 위험은 유출 직후에 끝나지 않는다

- **유출과 실제 탈취 사이의 시차는 예측할 수 없다.** 몇 시간 뒤일 수도, 몇 년 뒤일 수도 있다. 실제로 Snowflake 사건(2024)에서는 공격에 쓰인 계정 정보의 **인포스틸러 감염 시점이 탈취보다 약 4년 전**이었다. 콤보 리스트는 수년간 축적·유통된다.
- 유출이 공표된 직후에는 새 목록이 유통되고 보도로 표적이 알려져 위험이 올라가지만, **노출된 계정은 사용자가 비밀번호를 바꾸기 전까지 계속 위험하다.**
- 따라서 강화의 기준은 **시간이 아니라 노출 정황**이어야 한다. 다만 **노출 여부를 확신할 수는 없으므로**(1-6 참조) 이를 O/X 판정이 아니라 **위험 가중치**로 다룬다.

> **강화 강도:** 노출 정황의 확실성에 비례
**강화 해제:** 해당 계정의 **비밀번호 변경 또는 MFA 활성화** (일정 기간 경과가 아님)
> 

### 1-6. 노출 여부는 확신할 수 없다 (정직한 한계)

타사 유출 명단을 기업이 직접 받을 방법은 없다. KISA의 「털린 내 정보 찾기」도 **개인만 조회 가능**하며 기업에 명단을 제공하지 않는다. 따라서 “이 계정이 유출됐다”를 단정할 수 없고, **정황의 확실성에 따라 가중치를 다르게 주는 방식**을 택한다.

| 노출 판정 근거 | 확실성 | 가중치 |
| --- | --- | --- |
| 자사·계열 유출 사고 명단에 포함 | 높음 | 크게 |
| 다크웹 수집 크리덴셜과 일치 (위협 인텔리전스 연동) | 중간 | 중간 |
| 해당 계정에 스터핑 시도(반복 로그인 실패) 관측 | 낮음 | 작게 |
| 근거 없음 | — | 평시 기준 유지 |

> AhnLab도 대응 방안으로 **“다크웹 등에서 수집된 유출 인증정보로 블랙리스트를 구축하고, 해당 계정이 로그인에 사용되면 비밀번호 변경을 요구하거나 차단”**할 것을 권고한다. 다만 수집된 범위 안에서만 알 수 있어 누락이 존재한다.
> 

### 1-7. 기존 방식으로 왜 안 되나 (핵심)

유출이 알려지면 회사가 택할 수 있는 손쉬운 방법은 **넓게 거는 것**이다. CJ ONE이 연동 계정을 잠근 것이 그 예다.

이 방식이 왜 실패하는지 확인하기 위해, **실제 로그인 데이터 50만 건**에 가장 직관적인 룰을 걸어봤다. **“낯선 국가에서 낯선 기기로 로그인 성공”** - 계정탈취를 잡는 조건으로 누구나 먼저 떠올릴 규칙이다.

| 결과 | 값 |
| --- | --- |
| 잡아낸 계정탈취 | **11건 / 11건 (미탐 0)** |
| 발생한 경보 | **약 13만 건** |
| 전체 로그인 대비 | **약 26% — 로그인 4번 중 1번이 걸림** |
| 경보 중 진짜 탈취 비율 | **약 1만 2천 건에 1건** |

**즉 문제는 “탐지하지 못하는 것”이 아니라 “정상과 구분하지 못하는 것”이다.** 넓게 걸면 다 잡히지만 정상 이용자가 함께 막힌다. 우리가 풀어야 할 것은 탐지력이 아니라 **선별력**이다.

## 2. 목표 (Goal)

**제품 포지션:** **상시 작동하되, 유출 노출 정황이 있는 계정일수록 더 엄격하게** 판정한다. 
노출 여부를 확신할 수 없으므로 이를 게이트가 아닌 **위험 가중치**로 다룬다.

- 도용된 결제를 **승인 전에** 멈춘다. (승인 후에는 회수 불가 - 스타벅스는 사후 보전으로 800만 원을 부담했다)
- 탈취가 감지되면 **연결된 서비스까지 자동으로 함께 보호**한다. CJ ONE이 수동·전체로 한 일을 **자동·선별로** 한다.
- 노출 정황이 없는 이용자는 **평소 그대로** 결제한다. 강화는 **정황의 확실성에 비례**한다.
- **보호는 사용자가 조치할 때까지 유지한다.** 비밀번호 변경이나 MFA 활성화 전까지는 시간이 지나도 해제하지 않는다.
- 오탐 비용을 **차단이 아닌 인증 단계로 흡수**한다. → **"차단하는 FDS"가 아니라 "한 번 확인하는 FDS"**

---

- **MFA와의 관계**
    
    > 크리덴셜 스터핑의 가장 강력한 방어는 다단계 인증(MFA)이다. 하지만 **모든 결제에 MFA를 강제하면 이탈이 발생**하므로, 실제로는 선택 옵션으로 남고 정작 위험한 이용자는 켜지 않는다.
    **본 제품은 MFA를 대체하지 않는다. "언제 MFA를 요구할지 판단하는 층"이다.** 전부 걸면 이탈하고 안 걸면 뚫리므로, 위험할 때만 선별해서 건다.
    > 

## 3. 사용자 (User)

**주 사용자 — 김지현 (28세, 직장인, 서울)**

- 카카오 계정으로 여러 서비스에 로그인하고, 카카오페이에 카드 3장을 등록해뒀다. 주 2~3회 모바일 결제, 대부분 5만 원 이하.
- **연결된 서비스가 10곳이 넘지만 몇 개인지, 어디에 결제수단이 걸려 있는지 모른다.**
- 티빙 유출 문자를 받았지만 자기 카카오 계정이 어디까지 연결돼 있는지 몰라 그냥 넘겼다. **여러 사이트에 같은 비밀번호를 재사용**한다.
- **Pain:** 계정 하나가 뚫리면 어디까지 번지는지 감이 없다. 알림을 놓치면 피해가 그대로 확정된다.
- **Need:** 내가 아닐 때만 막아주고, **나일 때는 방해하지 말 것.** 문제가 생겼다면 **어디까지 영향받는지** 한눈에 보고 싶다.

**부 사용자 - 이상거래 운영 담당자**

- 하루 검토 가능량이 **500건 내외**로 한정돼 있다.
- **Need:** 위험 순으로 정렬된 목록과 **"왜 걸렸는지"**. 점수만 있으면 판단이 불가능하다.

**서비스 전제 (Constraint)**

- **SSO와 자체 간편결제를 함께 제공하는 플랫폼**을 전제한다. 그래야 로그인 신호와 결제를 같은 계정 키로 잇고, 연결 서비스 목록을 조회할 수 있다.
- 제3자 페이 연동만 하는 구조는 로그인 이벤트 접근이 불가해 **범위에서 제외**한다.

## 4. 핵심 기능 (Features)

### ① 노출 정황 점수화 - "얼마나 더 엄격하게 볼지 정한다"

- 계정별로 **유출 노출 정황을 점수로 산출**한다. 근거의 확실성에 따라 가중치가 다르다(1-6 표 참조).
    - 자사·계열 유출 명단 포함 → 크게 / 다크웹 수집 크리덴셜 일치 → 중간 / 스터핑 시도 관측 → 작게
- 이 점수는 **단독으로 조치를 유발하지 않는다.** ②③의 판정에 더해지는 **가중치**로만 쓰인다.
- **해제 조건은 시간 경과가 아니라 사용자의 조치**(비밀번호 변경 또는 MFA 활성화)다.
- **전면 잠금과의 차이:** CJ ONE은 연동 계정 전체를 **일괄 잠갔다.** 우리는 **정황에 비례해 감시 강도만 높이고** 결제는 계속 가능하게 한다.
- **왜 O/X가 아닌가:** 노출을 단정하면 판정이 틀렸을 때 진짜 위험한 계정을 놓친다. 가중치 방식은 근거가 없어도 평시 탐지가 계속 작동한다.

### ② 계정 장악 감지 - "알림을 끊으려는 시도를 잡는다" (S2)

- 로그인 성공 직후 **짧은 시간 내 이메일·비밀번호·연락처 변경**이 일어나면 장악 시도로 본다. 특히 **신규 기기에서** 발생하면 위험도가 급등한다.
- **조치:** 변경은 반영하되 **변경 전 원래 연락처로도 통보**하고, 이후 **N시간 동안 환금성 결제에 추가 인증**을 건다(쿨다운).
- 공격자가 알림 채널을 가로채도 **피해자에게 도달하는 경로를 하나 남긴다.**

### ③ 환금성 결제 위험 판정 - "마지막 방어선" (S4)

- 신호 조합: **신규 기기 + 낯선 위치 + 최근 계정 변경 + 기프티콘 등 환금성 카테고리 + 평소 대비 금액 이탈**
- 단일 신호로 판단하지 않는다. 검증 결과 **주소 국가 불일치 결제의 사기율은 12%**(전체 3.5% 대비 3.4배)였으나, 단독 사용 시 오탐이 과도했다.
- **판정 시점:** 카드사 승인 요청 **직전**. 승인 후에는 회수 불가.

**위험 수준별 조치 (사용자가 실제로 겪는 것)**

| 위험 | 사용자가 보는 화면 | 소요 시간 | 실패·무응답 시 |
| --- | --- | --- | --- |
| 낮음 | 아무것도 보이지 않음 
(정상 결제 완료) | 0초 | — |
| 중간 | "본인 확인이 필요해요" 
→ 생체인증 1회 | 3~5초 | 3회 실패 시 '높음'으로 상승 |
| 높음 | "확인 후 처리됩니다" 
+ 원래 연락처로 알림 발송 | 최대 10분 대기 | 10분 무응답 시 자동 취소(승인 아님) |
| 매우 높음 | "계정 보호를 위해 잠금되었습니다" | 즉시 | 신분 확인 절차로 이동 |

**설명 수위 원칙:** 판정 근거를 구체적으로 노출하지 않는다("해외 IP라서 막혔습니다" ✕). 공격자가 우회를 학습하기 때문. 대신 **무엇을 하면 풀리는지**는 명확히 안내한다.

### ④ 위험 전파 - "연결된 서비스까지 함께 보호한다" ★ SSO 특화

- ②·③에서 탈취가 감지되면, 해당 계정의 **연결 서비스 목록을 조회해 위험 신호를 전파**한다.
- **전파받은 서비스의 조치:** 해당 계정의 결제에 추가 인증 요구. 위험도가 매우 높으면 연결 일시 해제.
- **사용자 화면 - 영향 범위를 한눈에 보여준다**

> "낯선 기기에서 로그인이 감지되어 **연결된 서비스 12곳 중 결제수단이 등록된 5곳**의 결제를 일시 보호했습니다."
> 
> 
> → 서비스 목록 (각각 결제수단 등록 여부·마지막 로그인 표시)
> → `[모두 해제하고 비밀번호 변경]` / `[내가 맞습니다]`
> 
- **의도:** 사용자가 목록을 보며 "내 계정 하나에 이만큼 걸려 있었구나"를 자각하게 만들고, 그 자리에서 조치까지 연결한다. 23andMe 사례처럼 **연결 구조 자체가 위험을 증폭**시키기 때문이다.
- **CJ ONE 방식과의 차이:** 수동·전체 잠금 → **자동·위험 계정 한정·서비스별 차등**

### ⑤ 운영 검토 큐 - "사람이 최종 판단한다"

- 위험 점수 상위 K건을 **걸린 이유**(예: "발신≠수신 이메일, 신규 기기, 해외 국가코드")와 **권장 조치**와 함께 제공한다.
- 점수만 주면 담당자가 판단할 수 없다. **이유가 붙어야 몇 초 안에 결정**할 수 있다.

**Out of Scope (하지 않는 것)**

| 항목 | 제외 이유 |
| --- | --- |
| 전 계정 일괄 잠금 | 본 제품의 가설과 정반대 |
| 비밀번호 재사용 탐지 | **SSO도 제3자 사이트의 비밀번호는 알 수 없다.** 기술적으로 불가 |
| 상시 계정 보안 점검 화면 | 예방 영역으로 FDS 범위 밖. 본 제품의 화면은 **탐지 결과 통지**에 한정 |
| GPS 정밀 위치 수집 | 위치정보법상 별도 동의·사업신고 부담 → **IP 기반 추정으로 한정** |
| 조직형 공격 관계 분석(S5) | 타 이용자 정보 참조로 목적 외 이용 쟁점 → 법무 검토 후 차기 |

## 5. 성공 지표 (Metric)

**북극성 지표**

- **노출 계정군의 월간 부정결제 확정 피해 금액(원)** - 차지백·신고로 확정된 피해액 합계. 라벨이 늦게 확정되므로 **거래일 기준 T+30일까지 확정된 건**으로 집계한다.

**보조 지표**

- 승인 전 차단율 (확정 사기 중 승인 전에 막힌 비율)
- Recall - 미탐 최소화가 1순위
- Precision@K -검토 인력 500건 기준
- 위험 전파 도달률 - 탈취 감지 후 연결 서비스에 신호가 전달된 비율
- 사용자 응답률 - 보호 알림을 받고 조치(비밀번호 변경·연결 해제)를 실행한 비율

**성과를 어떻게 증명할 것인가 (측정의 난제)**

> FDS가 잘 작동하면 피해가 발생하지 않아 **"막아서 안 생긴 피해"를 직접 측정할 수 없다.** 홀드아웃(일부러 막지 않는 대조군)은 실제 고객 자산이 유출되므로 윤리적·법적으로 사용하기 어렵다.
**대안: 보류된 결제의 사후 확인 비율을 사용한다.** 보류 후 고객이 "내가 아님"으로 확인해준 건수를 **실제 예방된 피해**로 집계한다.
> 

**가드레일 지표 (넘으면 정책 롤백)**

| 지표 | 한계선 |
| --- | --- |
| 정상 결제 중 추가 인증·보류·차단을 겪은 비율 | ≤ 2.0% |
| 운영팀 일일 경보량 | ≤ 500건 |
| P95 조치 시간 | ≤ 30분 |
| 중간 위험 인증 통과 소요 시간 | ≤ 5초 |
| 위험 전파로 인한 정상 서비스 연결 해제 | ≤ 0.5% |

**대조 기준 (이번 가설의 핵심 검증)**

- 전면 잠금 방식 대비 **정상 결제 방해 건수**를 비교한다. 검증에서 전면 잠금형 룰은 경보 13만 건이었다. 목표는 **동일 차단율에서 경보량 1/10 이하.**

| 단계 | 공격자 행동 | 관찰 신호 | 우선순위 |
| --- | --- | --- | --- |
| S1 정찰 | 크리덴셜 대량 대입 | 한 IP에서 다수 계정 로그인 시도 + 높은 실패율 (공격 IP 중앙값 7개 계정, 정상 2개) | Should |
| S2 장악 | 이메일·연락처 변경으로 알림 차단 | 로그인 성공 → 단시간 내 민감정보 변경, 신규 기기 | **Must** |
| S3 테스트 | 소액 결제로 결제수단 유효성 확인 | 단시간 소액 연속 결제 후 금액 급증 | Could |
| S4 실행 | 기프티콘 결제 후 즉시 전송 | 신규 기기 + 낯선 위치 + 계정 변경 + 환금성 카테고리 + 금액 이탈 | **Must** |

## 부록 B. 알려진 한계 (Open Issues)

- **풍선효과** — 기프티콘 등 환금성 카테고리만 강하게 막으면 **공격이 다른 카테고리로 이동**할 수 있다. 그래서 카테고리는 단독 조건이 아닌 가중치로 쓰며, 이동 여부를 모니터링해 대상 카테고리를 확장하는 로드맵이 필요하다.
- **노출 판정의 불완전성** — 타사 유출 명단을 기업이 받을 수 없고, 다크웹 수집 정보도 수집된 범위에서만 유효하다. 노출 정황이 없어도 실제로는 노출된 계정이 존재한다. → 가중치 방식으로 완화하되 완전히 해소되지는 않는다.
- **공격자의 우회 가능성** — 최근 공격은 **헤드리스 브라우저**로 정상 트래픽을 위장하고, **프록시로 IP를 분산**하며, **User-Agent를 위조**한다. 국내 프록시를 경유하면 `is_foreign_ip`가, 기기 정보를 위조하면 `is_new_device`가 무력화될 수 있다. → 단일 신호에 의존하지 않고 **조합으로 판단**하며, 신호 재설계 주기를 정해야 한다.
- **개인정보 목적 외 이용** — 동일 IP의 타 계정 정보를 참조하는 관계 분석이 수집 목적 범위 내인지 법무 검토 필요.
- **라벨 지연** — 차지백 확정까지 2~8주. 피해자가 인지하지 못하면 사기가 정상으로 남는다(라벨 누락).
