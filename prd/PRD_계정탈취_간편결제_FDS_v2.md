# PRD: 계정 탈취 기반 부정결제 방어 FDS

작성: 송예원
버전: v2

**변경 요지**
1. 서비스 범위를 "SSO + 자체 간편결제를 한 회사가 함께 운영하는 플랫폼"으로 명시적으로 확정
2. 사용자가 위험의 심각성을 스스로 인지하게 만드는 요소 강화
3. 각 기능마다 지금 데이터로 검증 가능한 범위를 표기

**표기 규칙**
- `[확인]`: 출처가 있는 사실
- `[가정]`: 아직 검증되지 않은 전제
- `[데이터한계]`: 설계는 하되 이번 프로젝트에서 실증 불가

*FDS(Fraud Detection System, 이상거래 탐지 시스템): 결제·로그인 같은 거래 이벤트를 실시간으로 채점해 사기 의심 건을 차단하거나 보류하는 시스템.*

*SSO(Single Sign-On): 한 계정으로 여러 서비스에 로그인할 수 있는 방식. 카카오 계정으로 여러 앱에 가입·로그인하는 것이 대표적이다.*

---

## TL;DR

탈취된 계정으로 시도되는 환금성 결제(기프티콘, 충전금)를 승인 전에 멈추고, 같은 계정에 연결된 다른 서비스까지 자동으로 함께 보호하는 간편결제 FDS.

풀어 쓰면 이렇다.

- **누가**: 카카오 계정 하나로 여러 서비스에 로그인하고 카카오페이로 결제하는 일반 이용자
- **왜**: 계정이 뚫리면 기프티콘이 즉시 결제되고 전송되어 되돌릴 방법이 없고, 연결된 다른 서비스까지 동시에 위험해지기 때문
- **어떻게**: 로그인 신호(기기, 위치, 계정 변경)와 결제 신호(카테고리, 금액)를 하나의 계정 키로 이어 판정하고, 위험할 때만 한 번 더 확인시킨다
- **어떤 가치**: 평소에는 아무 방해가 없고, 위험할 때만 3초 인증 한 번으로 피해가 확정되기 전에 멈춘다

한 문장 포지션: "차단하는 FDS"가 아니라 "필요할 때만 한 번 확인하는 FDS".

## 1. Background: 왜 지금인가

### 1-1. 이미 벌어진 일이다 `[확인]`

| 사례 | 무슨 일이 있었나 |
| :--- | :--- |
| **스타벅스 코리아 (2023.7)** | 외부에서 확보한 아이디·비밀번호를 대입하는 **크리덴셜 스터핑**으로 해외 IP를 통해 앱에 부정 로그인. 고객 90여 명의 **충전금 약 800만 원**이 결제 도용. 회사가 IP 차단 후 **전액 보전** |
| **티빙 (2026.5)** | **1,953만 명**의 아이디·비밀번호·CI/DI·환불계좌 유출. 국내 유출 사고 중 4번째 규모 |
| **CJ ONE (2026.6~7)** | 티빙 유출 여파로 **연동 계정을 선제적으로 일괄 잠금**(6.10~7.2) |
| **23andMe (2023, 해외)** | 크리덴셜 스터핑으로 690만 명 정보 유출. 원인 중 하나가 "계정 하나가 침해되면 연결된 수백~수천 명까지 노출되는 연결 구조" |

*크리덴셜 스터핑(credential stuffing): 다른 곳에서 유출된 아이디·비밀번호 조합을 여러 서비스에 자동으로 대량 입력해 뚫리는 계정을 찾는 공격. "밀어넣는다(stuff)"는 이름 그대로다. 같은 비밀번호를 여러 사이트에 재사용하는 사람이 많아서 실제로 상당수가 뚫린다.*

스타벅스 사례가 이 프로젝트와 구조가 똑같다. 해외 IP, 크리덴셜 스터핑, 선불 자산 도용, 회사가 비용 부담. 그리고 대응이 전부 사후였다.

**보편성** `[확인]`: 개인정보보호위원회·KISA 「2024년 개인정보 유출 신고 동향」에서 크리덴셜 스터핑은 해킹 유형 중 **3번째로 많은 공격 기법**이며, Verizon 2024 DBIR은 도용된 인증정보 기반 공격이 **전체 침해의 약 3분의 1**을 차지한다고 보고했다.

### 1-2. 유출된 정보가 어떻게 결제까지 이어지나

**Q. 비밀번호는 암호화돼 있는데 어떻게 뚫리나?**

유출된 비밀번호는 보통 단방향 암호화(해시)되어 그 자체로는 쓸 수 없다. 티빙도 마찬가지였다. 그런데 공격자는 해시를 깰 필요가 없다. 과거 수많은 유출에서 이메일과 평문 비밀번호가 짝지어진 목록(**콤보 리스트**)이 이미 유통되고 있기 때문이다.

즉 이 공격은 "비밀번호를 알아내는" 것이 아니라 "이미 아는 조합을 밀어넣는" 것이다. 그래서 이름이 스터핑(밀어넣기, stuffing)이고, 비밀번호를 재사용한 계정이 뚫린다.

*해시(hash): 원문을 되돌릴 수 없는 문자열로 바꾸는 암호화. 저장된 해시만 있으면 원래 비밀번호를 알 수 없지만, 다른 곳에서 평문 비밀번호가 이미 새어 있다면 저장 방식이 무의미해진다.*

*콤보 리스트(combo list): 이메일 또는 아이디와 평문 비밀번호가 한 줄씩 짝지어진 대량의 유출 목록. 인터넷에 유통된다.*

**유출 항목이 각각 무슨 역할을 하나**

| 유출 항목 | 공격에서의 역할 |
| :--- | :--- |
| 이메일·아이디 | **표적 특정**: 누구를 공격할지 고른다 |
| 비밀번호(재사용 시) | **실제 침입 경로** (진짜 동력) |
| CI/DI | **확산 지도**: 서비스가 달라도 동일인 매칭이 가능해, 이 사람이 어느 서비스에 걸쳐 있는지 파악 |
| 환불계좌·연락처 | 피싱·사회공학 재료 |

CI는 침입 수단이 아니라 확산 지도다. CI로 로그인할 수는 없지만 "이 사람이 카카오에도 있다"를 알려준다. CJ ONE이 연동 계정을 잠근 근거도 이 연결 고리다.

*CI(Connecting Information): 통신사·본인확인기관이 발급하는 88바이트 문자열. 서비스가 달라도 같은 사람에게는 같은 값이 발급되므로 동일인 매칭에 쓰인다. 온라인 주민번호에 가까운 개념.*

*DI(Duplicated Information): 한 서비스 안에서 동일인 중복 가입을 막기 위한 값. 서비스마다 다르게 발급된다.*

### 1-3. 유출 출처에 따라 위협의 성격이 다르다

| 구분 | 공격자가 얻는 것 | 위협의 성격 |
| :--- | :--- | :--- |
| **A. 플랫폼 본체 유출** (카카오 자체) | 아이디 + 비밀번호(해시) + 계정 구조 | **연결된 모든 서비스가 동시 노출.** 단일 장애점이 터진 상황 |
| **B. 제3자 서비스 유출** (티빙 등) | 이메일·이름·CI (카카오 비밀번호는 안전) | CI로 표적을 특정해 밖에서 스터핑 시도. 외부 유입 공격 |

이 제품이 주로 다루는 것은 **B**다. A는 사고 대응(전사 강제 초기화) 영역이고, B는 상시 탐지로 걸러야 하는 영역이기 때문이다.

### 1-4. SSO 구조에서 피해가 연쇄된다

카카오 계정은 단일 장애점이다. 계정 하나가 뚫리면 그 계정으로 로그인한 연결 서비스 전체가 동시에 노출된다. 실제로 카카오는 `카카오계정 > 계정 이용 > 연결된 서비스 관리`에서 연결 서비스 목록과 해제 기능을 제공한다 `[확인]`. 즉 플랫폼은 어떤 서비스가 연결돼 있는지 이미 알고 있다.

계정 장악 후 도달 경로는 세 갈래다.

| 경로 | 설명 | 위험도 |
| :--- | :--- | :--- |
| **경로 ①: 앱 자체** | 카카오톡·카카오페이. 결제수단이 등록돼 있고 선물하기로 기프티콘 즉시 구매 가능 | 최고 |
| **경로 ②: 기존 연결 서비스** | 이미 로그인 전적이 있어 비밀번호 없이 줄줄이 통과 | 높음 |
| **경로 ③: 신규 가입** | 공격자가 새로 연결 | 낮음 |

### 1-5. 왜 하필 기프티콘인가: 회수 불가능성

이 제품의 가치는 "승인 전에 막는 것"인데, 그 가치의 크기는 상품마다 다르다.

| 상품 유형 | 사후 대응 가능성 | 실시간 차단의 가치 |
| :--- | :--- | :--- |
| 실물 배송 상품 | 배송 중단·회수 가능 | 상대적으로 낮음 |
| **기프티콘·충전금** | **승인 즉시 전송·사용, 회수 수단 없음** | **가장 높음** |

- 회수가 불가능한 영역이 곧 실시간 차단이 유일한 방어인 영역이다. 스타벅스가 충전금 800만 원을 전액 보전한 것도, 회수가 불가능해 보전 외에 방법이 없었기 때문이다.
- 오탐 비용도 줄어든다. 모든 결제에 엄격한 룰을 걸면 불편이 전면적으로 커지지만, 환금성 카테고리에 한정하면 영향 범위가 좁다.
- 공격자가 실제로 선호한다. 실물은 배송지가 필요해 추적되지만 기프티콘은 익명 전송·재판매가 가능하다.

단, 카테고리는 단독 조건이 아니라 위험 가중치로만 쓴다. 이 영역만 막으면 공격이 다른 카테고리로 이동하기 때문이다(Open Issues 풍선효과 참조).

### 1-6. 위험은 유출 직후에 끝나지 않는다

유출과 실제 탈취 사이의 시차는 예측할 수 없다. 몇 시간 뒤일 수도, 몇 년 뒤일 수도 있다. Snowflake 사건(2024)에서는 공격에 쓰인 계정 정보의 인포스틸러 감염 시점이 탈취보다 약 4년 전이었다 `[확인]`. 콤보 리스트는 수년간 축적·유통된다.

*인포스틸러(infostealer): 개인 PC에 몰래 설치돼 브라우저 저장 비밀번호·쿠키·자동입력 정보를 훔쳐가는 악성코드.*

따라서 강화의 기준은 "유출된 지 며칠 지났나"(시간)가 아니라 "이 계정이 노출됐을 정황이 있나"(상태)여야 한다.

- **강화 강도**: 노출 정황의 확실성에 비례
- **강화 해제**: 해당 계정의 **비밀번호 변경 또는 MFA 활성화** (시간 경과로는 풀리지 않는다)

*MFA(Multi-Factor Authentication, 다단계 인증): 비밀번호 외에 OTP나 생체인증 같은 추가 확인 수단을 요구하는 방식. 크리덴셜 스터핑의 가장 강력한 방어책이다.*

### 1-7. 그런데 노출 여부는 확신할 수 없다 (정직한 한계)

타사 유출 명단을 기업이 직접 받을 방법은 없다. KISA 「털린 내 정보 찾기」도 **개인만 조회 가능**하며 기업에 명단을 제공하지 않는다 `[확인]`. 따라서 "이 계정이 유출됐다"를 O/X로 단정할 수 없고, 정황의 확실성에 따라 가중치를 다르게 주는 방식을 택한다.

| 노출 판정 근거 | 확실성 | 가중치 | 이번 프로젝트 검증 여부 |
| :--- | :--- | :--- | :--- |
| 자사·계열 유출 사고 명단에 포함 | 높음 | 크게 | `[데이터한계]` 해당 데이터 없음 |
| 다크웹 수집 크리덴셜과 일치 (위협 인텔리전스 연동) | 중간 | 중간 | `[데이터한계]` 해당 데이터 없음 |
| 해당 계정에 스터핑 시도(반복 로그인 실패) 관측 | 낮음 | 작게 | 검증 가능 (RBA 로그인 데이터) |
| 근거 없음 | 없음 | 평시 기준 유지 | 해당 없음 |

*위협 인텔리전스(threat intelligence): 다크웹 등에서 유통되는 유출 크리덴셜, 공격자 인프라, 악성 도메인 같은 위협 정보를 상용으로 수집·판매하는 서비스. Recorded Future, Flashpoint, SpyCloud 등이 알려져 있다.*

AhnLab도 대응 방안으로 "다크웹 등에서 수집된 유출 인증정보로 블랙리스트를 구축하고, 해당 계정이 로그인에 사용되면 비밀번호 변경을 요구하거나 차단"할 것을 권고한다 `[확인]`. 다만 수집된 범위 안에서만 알 수 있어 누락이 존재한다.

### 1-8. 기존 방식으로 왜 안 되나: 이 프로젝트의 출발점

유출이 알려지면 회사가 택할 수 있는 손쉬운 방법은 넓게 거는 것이다. CJ ONE의 연동 계정 일괄 잠금이 그 예다.

이 방식이 왜 실패하는지 확인하려고, **실제 로그인 데이터 50만 건**에 가장 직관적인 룰을 걸어봤다. "낯선 국가에서 낯선 기기로 로그인 성공"이라는 조건이다. 계정 탈취를 잡으려 할 때 누구나 먼저 떠올릴 규칙이다.

| 결과 | 값 |
| :--- | :--- |
| 잡아낸 계정 탈취 | **11건 / 11건 (미탐 0)** |
| 발생한 경보 | **약 13만 건** |
| 전체 로그인 대비 | **약 26%. 로그인 4번 중 1번이 걸림** |
| 경보 중 진짜 탈취 비율 | **약 1만 2천 건에 1건** |

문제는 "탐지하지 못하는 것"이 아니라 "정상과 구분하지 못하는 것"이다. 우리가 풀어야 할 것은 탐지력이 아니라 **선별력**이다. 이 문장이 이 PRD 전체의 출발점이다.

## 2. Problem Statement

| 단계 | 내용 |
| :--- | :--- |
| **Current Situation** | 이용자는 카카오 계정 하나로 수십 개 서비스에 로그인하고, 그 계정에 카드가 등록돼 있다. 비밀번호는 여러 사이트에 재사용한다 |
| **Pain Point** | 타사에서 유출이 나도 내 카카오 계정이 위험한지, 어디까지 번지는지 알 방법이 없다. 알림을 놓치면 그대로 피해가 확정된다 |
| **Impact** | 기프티콘·충전금은 승인 즉시 사용되어 **회수가 불가능**하다. 결국 회사가 전액 보전하고(스타벅스 800만 원), 이용자는 계정 신뢰를 잃는다 |

**왜 단순 보안 강화가 아니라 제품 기회인가**

- 전부 막으면(전면 잠금) 정상 이용자 26%가 함께 막힌다. 사업이 불가능
- 아무것도 안 하면 피해가 확정된 뒤 보전 외에 방법이 없다
- 따라서 "누구를 언제 한 번 더 확인할지 고르는 판단 층"이 제품이 된다

**MFA와의 관계**

크리덴셜 스터핑의 가장 강력한 방어는 다단계 인증(MFA)이다. 하지만 모든 결제에 MFA를 강제하면 이탈이 발생하므로 실제로는 선택 옵션으로 남고, 정작 위험한 이용자는 켜지 않는다.

본 제품은 MFA를 대체하지 않는다. "언제 MFA를 요구할지 판단하는 층"이다. 전부 걸면 이탈하고 안 걸면 뚫리므로, 위험할 때만 선별해서 건다.

## 4. 서비스 전제 (Constraint): 왜 카카오·네이버 같은 구조여야 하나

**전제**: SSO(소셜 로그인)와 자체 간편결제를 한 회사가 함께 운영하는 플랫폼. 카카오(카카오톡·카카오페이·선물하기) 또는 네이버(네이버 로그인·네이버페이) 구조를 기준으로 한다.

**왜 이 구조여야만 하나: 세 가지 이유**

1. **같은 계정 키로 이을 수 있다.** 로그인 이벤트와 결제 이벤트가 같은 회사 안에 있어야 "이 로그인 뒤에 이 결제가 일어났다"를 판정할 수 있다. 이 제품의 모든 기능이 여기에 의존한다.
2. **연결 서비스 목록을 조회할 수 있다.** 위험 전파(기능 ④)는 "이 계정에 어떤 서비스가 붙어 있나"를 알아야 성립한다. SSO 제공자만 이 목록을 갖고 있다.
3. **통신 비용이 발생하지 않는다.** 제3자 페이만 연동하는 구조라면, 결제 판정에 필요한 로그인 신호를 결제 건마다 외부로 넘겨야 한다. 실시간 결제 승인 경로에 외부 호출이 끼면 지연·장애·비용이 모두 커지고, 개인정보 제3자 제공 이슈까지 생긴다. 같은 회사 안에서는 이 비용이 사실상 0이다.

**범위에서 제외**: 제3자 페이 연동만 하는 구조 (로그인 이벤트 접근 불가, 통신 비용 과다)

## 6. Key Scenario: 김지현의 하루

1. **Context**: 지현은 3개월 전 티빙에 가입할 때 카카오 계정과 같은 비밀번호를 썼다. 티빙 유출 문자를 받았지만 넘겼다
2. **Problem Moment**: 새벽 2시, 해외 IP와 처음 보는 안드로이드 기기에서 지현의 카카오 계정 로그인이 성공한다. 공격자는 곧바로 가입 이메일을 자기 주소로 바꾸고, 선물하기에서 10만 원권 백화점 상품권 5장을 결제하려 한다
3. **Product Interaction**:
   - 로그인 직후 이메일 변경이 감지되어 **장악 시도**로 판정. 변경은 반영하되 원래 이메일로도 통보, 이후 환금성 결제에 추가 인증 부과
   - 결제 시점에 `신규 기기 + 해외 위치 + 방금 계정 변경 + 환금성 카테고리 + 평소 5만 원의 10배`가 겹쳐 '**매우 높음**' 판정. 승인 요청 직전 차단
   - 동시에 연결 서비스 12곳 중 결제수단이 등록된 5곳에 위험 신호 전파
4. **Resolution**: 아침에 지현이 알림을 연다. "연결된 서비스 12곳 중 5곳에 결제수단이 등록돼 있었고, 지난밤 그중 3곳에서 결제 시도가 있었습니다." 목록을 보며 처음으로 자기 계정의 영향 범위를 자각하고, `[모두 해제하고 비밀번호 변경]`을 누른다
5. **Outcome**: 피해 0원. 지현은 비밀번호를 바꾸고 MFA를 켠다. 강화 상태 해제, 다음날부터 평소대로 결제

### 기능 ①. 노출 정황 점수화: "얼마나 더 엄격하게 볼지 정한다"

**하는 일**: 계정별로 유출 노출 정황을 점수로 산출한다. 근거의 확실성에 따라 가중치가 다르다(1-7 표).

**핵심 설계 판단 두 가지**

- 단독으로 조치를 유발하지 않는다. ②③의 판정에 더해지는 **가중치**로만 쓴다
- **왜 O/X가 아닌가**: 노출을 단정하면 판정이 틀렸을 때 진짜 위험한 계정을 놓친다. 가중치 방식은 근거가 없어도 평시 탐지가 계속 작동한다

**해제 조건**: 시간 경과가 아니라 사용자의 조치 (비밀번호 변경 또는 MFA 활성화)

**전면 잠금과의 차이**: CJ ONE은 연동 계정 전체를 일괄 잠갔다. 우리는 정황에 비례해 감시 강도만 높이고 결제는 계속 가능하게 한다.

`[데이터한계]` **검증 범위**: 3가지 근거 중 스터핑 시도 관측(한 IP에서 여러 계정에 반복 로그인 실패)만 공개 데이터로 계산 가능하다. 자사 유출 명단과 다크웹 수집 크리덴셜은 기업 내부 자산 또는 유료 위협 인텔리전스 계약이 있어야 접근 가능해서 어떤 공개 데이터셋에도 존재하지 않는다.

**대응**: 노출 점수를 임의로 부여한 가상 그룹을 만들어, "노출 점수가 있는 그룹에 임계값을 낮추면 탐지율과 경보량이 각각 얼마나 변하는가"를 **민감도 분석**으로 제시한다. 실측이 아니라 시뮬레이션임을 발표에서 명시한다.

### 기능 ②. 계정 장악 감지: "알림을 끊으려는 시도를 잡는다" (공격 단계 S2)

**하는 일**: 로그인 성공 직후 짧은 시간 내에 이메일·비밀번호·연락처가 변경되면 장악 시도로 본다. 특히 신규 기기에서 발생하면 위험도가 급등한다.

**왜 이게 핵심인가: 공격자의 첫 행동이기 때문**

공격자가 계정을 뺏은 뒤 가장 먼저 하는 일은 결제가 아니다. **알림 채널을 끊는 것**이다. 이메일과 휴대폰 번호를 자기 것으로 바꿔놓으면, 이후 어떤 결제를 해도 피해자에게 아무 통지가 가지 않는다. 이 시점을 잡으면 결제가 일어나기 전에 개입할 수 있다.

**조치**

- 변경은 반영하되 변경 전 원래 연락처로도 통보한다. 공격자가 알림 채널을 가로채도 피해자에게 도달하는 경로를 하나 남긴다
- 이후 N시간 동안 환금성 결제에 추가 인증을 건다 (쿨다운)

`[데이터한계]` **검증 범위**: 공개 로그인 데이터셋(RBA)에는 로그인 이벤트만 있고 계정 정보 변경 이벤트 자체가 기록되지 않는다. 즉 이 기능은 설계는 완결돼 있으나 이번 프로젝트에서 모델로 학습·평가할 수 없다.

**대응**: 탐지 룰(로그인 성공 후 T분 내 민감정보 변경 여부, 신규 기기 여부)을 규칙 명세로 문서화하고, 실제 운영 데이터에서 튜닝해야 할 파라미터(T값, 쿨다운 N시간)를 오픈 이슈로 남긴다. 카카오·네이버 내부에는 이 로그가 반드시 존재하므로 실무 적용 시에는 즉시 구현 가능한 기능이다.

---

### 기능 ③. 환금성 결제 위험 판정: "마지막 방어선" (공격 단계 S4)

**하는 일**: 승인 요청 직전에 결제의 위험도를 판정한다. 승인 후에는 회수가 불가능하므로 시점이 결정적이다.

**신호 조합** (단일 신호로 판단하지 않는다)

`신규 기기` + `낯선 위치` + `최근 계정 변경` + `기프티콘 등 환금성 카테고리` + `평소 대비 금액 이탈`

검증 결과 주소 국가 불일치 결제의 사기율은 12%로 전체 평균 3.5%의 3.4배였으나 `[확인]`, 단독 사용 시 오탐이 과도했다. 조합해야 선별력이 나온다.

**위험 수준별 조치 (사용자가 실제로 겪는 것)**

| 위험 | 사용자가 보는 화면 | 소요 시간 | 실패·무응답 시 |
| :--- | :--- | :--- | :--- |
| **낮음** | 아무것도 보이지 않음 (정상 결제 완료) | 0초 | 해당 없음 |
| **중간** | "본인 확인이 필요해요" 후 생체인증 1회 | 3~5초 | 3회 실패 시 '높음'으로 상승 |
| **높음** | "확인 후 처리됩니다" + 원래 연락처로 알림 | 최대 10분 대기 | 10분 무응답 시 자동 취소 (승인 아님) |
| **매우 높음** | "계정 보호를 위해 잠금되었습니다" | 즉시 | 신분 확인 절차로 이동 |

**설명 수위 원칙**: 판정 근거를 구체적으로 노출하지 않는다("해외 IP라서 막혔습니다"는 금지). 공격자가 우회를 학습하기 때문. 대신 무엇을 하면 풀리는지는 명확히 안내한다.

`[데이터한계]` **검증 범위: 이번 프로젝트의 주력**

결제 데이터(IEEE-CIS)로 대부분 구현 가능하다. 단 두 가지 보정이 필요하다.

- **'기프티콘' 카테고리가 없다**: `ProductCD` 중 디지털 성격 코드를 환금성 대리 지표로 지정하고, 그 선택 근거를 문서화한다
- **사용자 ID가 없다**: '평소 대비 금액 이탈'을 계산하려면 같은 사람의 거래를 묶어야 하는데 원본에 user_id가 없다. `card1 + addr1 + 등록일 역산`으로 UID를 복원해야 한다. 이 작업이 ③의 선행 조건이다

*IEEE-CIS Fraud Detection: 미국 이커머스 결제 회사 Vesta가 공개한 결제 데이터셋. 캐글에 공개돼 있고 컬럼명 대부분이 익명화(V1~V339)돼 있다. 사기 라벨(isFraud)이 붙어 있어 지도학습에 쓴다.*

*ProductCD: IEEE-CIS 데이터셋의 상품 코드 컬럼. C, H, R, S, W 다섯 개 값을 가지며 각각이 어떤 상품 유형인지는 공개돼 있지 않다. 이번 프로젝트에서 어느 코드를 환금성으로 볼지 별도로 판단해야 한다.*

---

### 기능 ④. 위험 전파: "연결된 서비스까지 함께 보호한다" (SSO 특화 기능)

**하는 일**: ②·③에서 탈취가 감지되면, 해당 계정의 연결 서비스 목록을 조회해 위험 신호를 전파한다.

**전파받은 서비스의 조치**: 해당 계정의 결제에 추가 인증 요구. 위험도가 매우 높으면 연결 일시 해제.

**CJ ONE 방식과의 차이**: 수동·전체 잠금이 아니라 자동·위험 계정 한정·서비스별 차등

`[데이터한계]` **검증 범위**: "어떤 계정이 어떤 서비스에 연결돼 있는가"라는 데이터는 SSO 제공자만 보유하며, 개인정보보호법상 외부 공개가 불가능하다. 공개 데이터셋으로 대체할 방법이 없다.

**대응**: 전파 규칙(누구에게, 무엇을, 얼마 동안)을 명세로 정의하고, 가상의 연결 서비스 프로필을 만들어 화면 프로토타입으로 시연한다. 이 기능은 "카카오가 이미 보유한 데이터 구조를 전제로 한 설계"임을 발표에서 명시한다.

---

### 기능 ⑤. 사용자 인지 화면: "내 계정에 이만큼 걸려 있었구나" (v2 신규 강화)

기존 PRD에서 ④의 하위 요소였던 것을 독립 기능으로 승격했다. 이유는 페르소나 김지현의 진짜 Pain이 "막히는 것"이 아니라 "위험을 인지하지 못하는 것"이기 때문이다. 티빙 유출 문자를 받고도 넘긴 이유는 무관심이 아니라, 그 문자가 자기에게 무엇을 의미하는지 몰랐기 때문이다.

**심각성 인지를 만드는 네 가지 요소**

**요소 1. 추상적 경고 대신 구체적 숫자**

| 기존 방식 | 우리 방식 |
| :--- | :--- |
| "계정 보안에 유의하세요" | "연결된 서비스 12곳 중 결제수단이 등록된 5곳의 결제를 일시 보호했습니다" |
| "비정상 접근이 감지되었습니다" | "지난밤 3곳에서 결제 시도가 있었고 모두 차단했습니다" |

숫자가 있어야 "남 일"이 "내 일"이 된다.

**요소 2. 영향 범위 목록: 자기 계정의 크기를 처음 보게 한다**

각 연결 서비스마다 다음을 표시한다.

- 서비스명
- 결제수단 등록 여부 (등록됨 / 없음)
- 마지막 로그인 시점
- 지난밤 시도 여부 (시도 있었음 / 없음)

대부분의 이용자는 자기 계정에 몇 개 서비스가 붙어 있는지 모른다. 목록 자체가 가장 강력한 인지 장치다.

**요소 3. 막힌 피해를 금액으로 환산해 보여준다**

> "차단된 결제 3건, 합계 47만 원. 기프티콘은 승인되면 즉시 사용되어 되돌릴 수 없습니다."

FDS의 근본적 딜레마는 잘 작동할수록 아무 일도 안 일어난 것처럼 보인다는 것이다. 막은 피해를 금액으로 보여줘야 사용자가 가치를 인식하고, 후속 조치(비밀번호 변경)로 이어진다.

**요소 4. 인지에서 조치까지 한 화면에서 끝낸다**

```
[모두 해제하고 비밀번호 변경]   (주 버튼)
[내가 맞습니다]                 (오탐이었을 때)
```

인지시켜 놓고 "설정에서 변경하세요"라고 하면 대부분 이탈한다. 자각한 순간이 조치 전환율이 가장 높은 순간이다.

`[데이터한계]` 화면 프로토타입으로 구현한다. 실제 연결 서비스 데이터가 필요한 부분은 가상 프로필로 대체한다.

---

### 기능 ⑥. 운영 검토 큐: "사람이 최종 판단한다"

- 위험 점수 상위 K건을 걸린 이유(예: "발신≠수신 이메일, 신규 기기, 해외 국가코드")와 권장 조치와 함께 제공한다
- 점수만 주면 담당자가 판단할 수 없다. 이유가 붙어야 몇 초 안에 결정할 수 있다
- K = 500 (담당자 하루 검토 가능량)

*SHAP(SHapley Additive exPlanations): 머신러닝 모델이 어떤 예측을 왜 그렇게 했는지를 각 입력 변수의 기여도로 분해해 보여주는 기법. LightGBM 같은 트리 모델과 궁합이 좋다.*

**검증 범위**: 결제 모델에 SHAP을 붙여 건별 기여 요인을 한국어 문장으로 출력. 구현 가능.

## 9. Scope

### In Scope

| 구분 | 항목 |
| :--- | :--- |
| **Must** | ② 계정 장악 감지 (S2), ③ 환금성 결제 위험 판정 (S4), ⑤ 사용자 인지 화면 |

| 항목 | 제외 이유 |
| :--- | :--- |
| 전 계정 일괄 잠금 | 본 제품의 가설과 정반대 |

## 10. 데이터 검증 범위 총괄 (v2 신규)

**이 표가 v2의 핵심이다.** 기능은 6개 모두 설계하되, 공개 데이터로 실증 가능한 범위를 기능별로 명시한다.

| 기능 | 설계 | 공개 데이터 검증 | 사용 데이터 | 못 하는 이유 |
| :--- | :---: | :---: | :--- | :--- |
| ① 노출 정황 점수화 | 완결 | 부분 | RBA 로그인 | 3개 근거 중 2개(유출 명단·다크웹)가 기업 내부 자산 |
| ② 계정 장악 감지 | 완결 | 불가 | 없음 | 계정 변경 이벤트 로그가 공개 데이터에 없음 |
| ③ 환금성 결제 판정 | 완결 | 가능 (주력) | IEEE-CIS 결제 | 카테고리·UID만 근사 처리 |
| ④ 위험 전파 | 완결 | 불가 | 없음 | SSO 연결 서비스 목록은 외부 공개 불가 |
| ⑤ 사용자 인지 화면 | 완결 | 프로토타입 | 가상 프로필 | 위와 동일 |
| ⑥ 운영 검토 큐 | 완결 | 가능 | IEEE-CIS + SHAP | 해당 없음 |

**두 데이터를 하나로 합칠 수 없는 이유**

로그인 데이터(RBA, 노르웨이 대학 서비스)와 결제 데이터(IEEE-CIS, 미국 이커머스)는 겹치는 사용자가 없다. 따라서 "로그인 위험 점수를 결제 판정에 넣는다"는 파이프라인을 실제로 돌릴 수 없다.

이건 데이터를 바꿔서 해결되는 문제가 아니다. SSO 로그인 + 결제 + 유출 이력이 한 사람 키로 묶인 데이터는 카카오·네이버 내부에는 존재하지만, 개인정보보호법상 외부 공개가 불가능하다. IEEE-CIS가 컬럼명을 V1~V339로 익명화해 공개된 것도 같은 이유다.

*RBA(Risk-Based Authentication) Login Dataset: Wiefling 외(2022)가 공개한 로그인 이벤트 데이터셋. 3천만 건의 로그인 기록에 공격 IP 여부와 계정 탈취 여부 라벨이 붙어 있다. 노르웨이 대학 서비스의 실제 로그를 익명화해 배포됐다.*

**발표 프레이밍**:

> "이 설계는 카카오가 실제로 보유한 데이터 구조를 전제로 합니다. 다만 해당 데이터는 외부 공개가 불가능하므로, 본 프로젝트에서는 로그인 단계와 결제 단계를 각각 공개 데이터셋으로 개별 검증하고, 두 단계를 잇는 가중치 효과는 민감도 분석으로 제시합니다."

"데이터가 없어서 못 했다"가 아니라 **"실무에서는 되는 구조인데 공개 데이터의 한계로 이렇게 나눴다"**가 된다.

**로그인 데이터의 라벨 문제** `[결정 필요]`

RBA 데이터셋의 `Is Account Takeover` 양성은 11건뿐이다. 이걸 라벨로 지도학습을 돌릴 수 없다. 두 선택지가 있다.

| 선택지 | 방식 | 장단점 |
| :--- | :--- | :--- |
| **A. 비지도 이상탐지** | Isolation Forest로 라벨 없이 이상 로그인 탐지, 11건이 상위 몇 %에 들어오는지로 평가 | 수업에서 배운 도구와 정합. 성능 지표가 약함 |
| **B. `Is Attack IP`를 라벨로** | 지도학습(LightGBM) 가능 | 양성 충분. 문제가 "탈취 탐지"가 아닌 "공격 IP 탐지"로 바뀜 |

*Isolation Forest: 라벨 없이 이상치를 찾는 비지도학습 알고리즘. 데이터를 무작위로 잘라나갔을 때 몇 번 만에 혼자 남는지를 기준으로 이상 정도를 잰다. 뭉쳐 있는 데이터는 여러 번 잘라야 분리되고, 동떨어진 데이터는 몇 번만 잘라도 분리된다는 성질을 이용한다.*

*LightGBM: 마이크로소프트가 공개한 그래디언트 부스팅 트리 라이브러리. 지도학습에 쓰며, 결측치 처리와 학습 속도가 빨라서 캐글 등에서 많이 쓴다.*

## 11. UX Principles

| 항목 | 원칙 |
| :--- | :--- |
| **코어 감정** | 불안에서 통제감으로. "막혔다"가 아니라 "내가 지켜지고 있다" |

| **금지 표현** | 판정 근거의 구체적 노출("해외 IP라서 차단"). 공격자 우회 학습 유발 |

| **대기 시간** | 중간 위험 인증은 5초 이내. 이 시간을 넘기면 정상 이용자가 이탈한다 |

## 12. Success Metrics

### 북극성 지표

**노출 계정군의 월간 부정결제 확정 피해 금액(원)**

차지백·신고로 확정된 피해액 합계. 라벨이 늦게 확정되므로 거래일 기준 T+30일까지 확정된 건으로 집계한다.

*이 지표를 고른 이유*: TL;DR의 가치("회수 불가능한 피해가 확정되기 전에 멈춘다")와 직접 연결되고, 탐지 건수 같은 허영 지표가 아니라 실제로 막힌 돈을 센다.

*차지백(chargeback): 카드 소지자가 부정 사용이라고 신고했을 때 카드사가 가맹점 결제를 강제로 취소하는 절차. 확정까지 보통 2~8주가 걸린다.*

### 보조 지표

| 지표 | 정의 | 북극성과의 연결 |
| :--- | :--- | :--- |
| 승인 전 차단율 | 확정 사기 중 승인 전에 막힌 비율 | 승인 후 차단은 피해액을 못 줄인다 |
| Recall | 미탐 최소화가 1순위 | 놓친 건이 곧 피해액 |
| Precision@500 | 검토 인력 500건 기준 | 운영 부하가 한계를 넘으면 시스템이 멈춘다 |
| 위험 전파 도달률 | 탈취 감지 후 연결 서비스에 신호가 전달된 비율 | 2차 피해 차단 |
| 사용자 응답률 | 보호 알림 후 실제 조치(비번 변경·연결 해제) 실행 비율 | 기능 ⑤의 직접 성과 지표 |

*Recall(재현율): 실제 사기 중에 시스템이 사기로 판정한 비율. 놓친 건이 곧 피해로 이어지므로 FDS에서 우선순위가 높다.*

*Precision(정밀도): 시스템이 사기라고 판정한 것 중 실제 사기인 비율. 오탐이 많으면 이 값이 떨어지고 운영 부담이 커진다.*

### 성과를 어떻게 증명할 것인가 (측정의 난제)

FDS가 잘 작동하면 피해가 발생하지 않아 **"막아서 안 생긴 피해"를 직접 측정할 수 없다.** 홀드아웃(일부러 막지 않는 대조군)은 실제 고객 자산이 유출되므로 윤리적·법적으로 사용하기 어렵다.

**대안**: 보류된 결제의 사후 확인 비율을 사용한다. 보류 후 고객이 "내가 아님"으로 확인해준 건수를 실제 예방된 피해로 집계한다.

*홀드아웃(holdout): 정책·모델의 효과를 검증하기 위해 일부 트래픽에는 정책을 적용하지 않고 원래 상태 그대로 두는 대조군. A/B 테스트의 B 그룹에 가깝다.*

### 가드레일 지표 (넘으면 정책 롤백)

| 지표 | 한계선 |
| :--- | :--- |
| 정상 결제 중 추가 인증·보류·차단을 겪은 비율 | 2.0% 이하 |
| 운영팀 일일 경보량 | 500건 이하 |
| P95 조치 시간 | 30분 이하 |
| 중간 위험 인증 통과 소요 시간 | 5초 이하 |
| 위험 전파로 인한 정상 서비스 연결 해제 | 0.5% 이하 |

*P95: 상위 5%를 뺀 나머지 95%의 최댓값. "가장 오래 걸린 5% 케이스를 제외했을 때 최악의 소요 시간"을 뜻한다. 평균보다 실제 사용자 경험을 잘 대변한다.*

### 대조 기준 (이번 가설의 핵심 검증)

전면 잠금 방식 대비 정상 결제 방해 건수를 비교한다. 1-8 검증에서 전면 잠금형 룰은 경보 13만 건이었다.

**목표: 동일 차단율에서 경보량 1/10 이하 (13만 건 → 1.3만 건 이하)**

## 13. Constraints & Assumptions

### Constraints (바꿀 수 없는 것)

- SSO와 자체 간편결제를 같은 회사가 운영하는 구조여야 한다
- 타사 유출 명단은 기업이 받을 수 없다 (KISA 「털린 내 정보 찾기」는 개인 조회만)
- GPS 정밀 위치는 위치정보법상 부담. IP 기반 추정만
- 차지백 확정까지 2~8주. 라벨 지연
- 이번 프로젝트에서는 로그인·결제 데이터가 서로 다른 출처라 통합 파이프라인 실증 불가

### Assumptions (검증해야 할 것)

| 가정 | 검증 방법 |
| :--- | :--- |
| 신호를 조합하면 단일 룰 대비 경보량을 1/10로 줄이면서 탐지율을 유지할 수 있다 | IEEE-CIS에서 단일 룰 vs 조합 모델 성능 비교 |
| 노출 정황 가중치를 넣으면 같은 경보량에서 탐지율이 오른다 | 가상 노출군 대상 민감도 분석 |
| 사용자는 영향 범위 목록을 보면 실제로 조치한다 | `[미검증]` 사용자 응답률 측정 필요 |
| 환금성 카테고리 한정이 오탐 비용을 실제로 줄인다 | `ProductCD` 별 사기율·오탐율 비교 |

## 부록 A. 공격 단계별 탐지 시나리오

| 단계 | 공격자 행동 | 관찰 신호 | 우선순위 | 검증 |
| :--- | :--- | :--- | :---: | :---: |
| **S1 정찰** | 크리덴셜 대량 대입 | 한 IP에서 다수 계정 로그인 시도 + 높은 실패율 (공격 IP 중앙값 7개 계정, 정상 2개) | Should | 가능 |
| **S2 장악** | 이메일·연락처 변경으로 알림 차단 | 로그인 성공 → 단시간 내 민감정보 변경, 신규 기기 | **Must** | 불가 |
| **S3 테스트** | 소액 결제로 결제수단 유효성 확인 | 단시간 소액 연속 결제 후 금액 급증 | Could | 가능 |
| **S4 실행** | 기프티콘 결제 후 즉시 전송 | 신규 기기 + 낯선 위치 + 계정 변경 + 환금성 카테고리 + 금액 이탈 | **Must** | 가능 |
| **S5 확산** | 여러 탈취 계정을 한 곳으로 수렴 | 서로 다른 계정의 결제가 동일 수취인·기기로 집중 | Could | 불가 (법무) |

## 부록 B. 알려진 한계

- **풍선효과**: 기프티콘 등 환금성 카테고리만 강하게 막으면 공격이 다른 카테고리로 이동할 수 있다. 그래서 카테고리는 단독 조건이 아닌 가중치로 쓰며, 이동 여부를 모니터링해 대상 카테고리를 확장하는 로드맵이 필요하다.
- **노출 판정의 불완전성**: 타사 유출 명단을 기업이 받을 수 없고, 다크웹 수집 정보도 수집된 범위에서만 유효하다. 노출 정황이 없어도 실제로는 노출된 계정이 존재한다.
- **공격자의 우회 가능성**: 최근 공격은 헤드리스 브라우저로 정상 트래픽을 위장하고, 프록시로 IP를 분산하며, User-Agent를 위조한다. 국내 프록시를 경유하면 `is_foreign_ip`가, 기기 정보를 위조하면 `is_new_device`가 무력화될 수 있다. 단일 신호에 의존하지 않고 조합으로 판단하며, 신호 재설계 주기를 정해야 한다.

*헤드리스 브라우저(headless browser): 화면 표시 없이 자동화 스크립트로만 작동하는 브라우저. 정상 사용자 트래픽처럼 보이지만 실제로는 봇이다.*

*User-Agent: 브라우저나 앱이 서버에 접속할 때 자기 자신을 알리는 문자열(예: "Chrome 120 on Windows 11"). 서버는 이 정보를 기기 판별에 쓰는데, 위조가 쉬워서 단독 신호로는 신뢰도가 낮다.*

- **개인정보 목적 외 이용**: 동일 IP의 타 계정 정보를 참조하는 관계 분석이 수집 목적 범위 내인지 법무 검토 필요.
- **라벨 지연**: 차지백 확정까지 2~8주. 피해자가 인지하지 못하면 사기가 정상으로 남는다 (라벨 누락).
