# PRD: 개인정보 유출 이후 이커머스 부정결제 탐지 FDS

> Groom Fintech 6기, FDS 팀플 (4조), 작성자: 송예원
> 최종 산출물은 팀 GitHub 저장소의 개인 브랜치에 병합한다.

---

## TL;DR

대규모 개인정보 유출로 크리덴셜이 이미 탈취된 상황에서, **이커머스 간편결제로 등록된 결제수단을 도용당할 위험에 놓인 일반 소비자**가, 로그인과 결제 두 단계의 실시간 위험 판단을 통해 **계정 탈취와 부정결제를 사전에 차단하거나 추가 인증으로 걸러내어**, 돈을 잃지 않고 정상 거래는 방해받지 않는 경험을 얻는다.

> 이 문서는 수업 워크시트(50~52회차) 기반 학습용 PRD다. 시장과 수치 항목 중 검증되지 않은 부분은 `(가정, 검증 필요)`로 표기했다. 모델 성능 수치는 RBA와 IEEE-CIS 공개 데이터로 팀이 직접 학습하고 평가한 실측값이다.

## Background

- 2026년 5월 티빙 개인정보 유출로 1,953만 명의 아이디, 비밀번호, CI, DI, 환불 계좌번호가 유출됐다. (출처: 50회차 워크시트 사건 정리, 언론 보도)
- CI는 온라인상 주민등록번호 역할을 하는 값이라 다른 서비스의 가입 여부까지 추적 가능하며, 유출된 크리덴셜을 다른 사이트에 대입하는 **크리덴셜 스터핑** 공격으로 2차 피해가 확산된다. (출처: 50회차 워크시트)
- 티빙은 사고 직후 CJ ONE 연동 계정을 전면 잠그는 초강수를 뒀다. 이는 **오탐률 100%를 감수한 조치**로, 정상 사용자 전원을 위험으로 판정한 것이다. 위험 신호가 있는 거래만 선별 차단하는 FDS가 있었다면 불필요한 전면 잠금을 피할 수 있었다. (출처: 50회차 워크시트 분석)
- 이커머스 간편결제(네이버페이, 카카오페이, 쿠팡페이 등)는 계정에 카드와 계좌가 등록돼 있어, 계정이 뚫리면 결제수단이 통째로 노출된다. 유출 사고가 이커머스와 페이 영역의 실제 금전 피해로 번지는 경로가 형성돼 있다.

## Problem Statement

- **Current Situation:** 소비자는 한 번 등록한 결제수단으로 간편하게 결제한다. 여러 서비스에 같은 비밀번호를 재사용하는 경우가 많다.
- **Pain Point:** 외부 유출로 크리덴셜이 털리면, 공격자가 크리덴셜 스터핑으로 로그인한 뒤 이메일과 비밀번호를 변경하고, 등록된 결제수단으로 고액 결제 또는 상품권 구매로 현금화한다. 피해자는 알림도 못 받고(이메일이 변경됨), 카드 명세서를 보고서야 알게 된다.
- **Impact:** 승인 즉시 피해가 확정되며(특히 상품권은 즉시 사용되고 전송돼 회수 불가), 피해자는 돈과 계정 통제권을 모두 잃는다. 회사는 차지백 비용과 CS 부담을 지고, 대규모 발생 시 "○○페이가 뚫렸다"는 신뢰 붕괴로 이어진다.
- **근거:**
  - 크리덴셜 스터핑은 실측으로 확인된다. RBA 로그인 데이터(50만 건 샘플)에서 공격 IP는 계정을 훨씬 많이 두드린다. 공격 IP의 계정 수 중앙값 7개(정상 2개, 최대 268개). (RBA 실측)
  - 결제 층 사기율은 IEEE-CIS 기준 3.5%. 정상 거래가 사기보다 약 28배 많아 단순 룰로는 오탐이 폭증하는 구조. (IEEE-CIS 실측)
  - 실제 유출 후 계정 도용에서 결제 피해로 이어진 사례가 보도된 바 있음. (가정, 검증 필요)

## Target Persona

- **이름과 상황:** 김도현(가명), 29세, 직장인, 서울 거주. 네이버페이와 쿠팡페이에 신용카드를 등록해두고 평소 모바일로 3만 원대 소액 결제를 자주 한다. 여러 사이트에 비슷한 비밀번호를 재사용해왔다.
- **Pains:** 어느 사이트에서 내 정보가 털렸는지 알 수 없다. 결제 알림을 놓치면 도용을 뒤늦게 안다. 그렇다고 결제할 때마다 인증을 여러 번 하는 건 번거롭다.
- **Needs:** 내가 하는 정상 결제는 평소처럼 매끄럽게, 남이 도용한 결제는 확실히 막아주기를 원한다. 막을 때도 무작정 차단이 아니라 본인 확인 한 번으로 풀 수 있기를 바란다.
- **MVP 첫 고객 선정 이유:** 간편결제에 결제수단을 등록해둔 활성 사용자층은 시장성이 크고, 유출 사고 직후 피해가 집중되는 시급성이 높으며, 계정 탈취 탐지 로직은 타 서비스로 확장 가능하다.

---

## Goals

- **User Goal:** 도용당한 결제는 사전에 막고, 내 정상 결제는 방해받지 않는다. 막힐 때도 본인 확인 한 번으로 즉시 진행할 수 있다.
- **Product Goal:** "차단만 하는 FDS"가 아니라 "로그인과 결제 두 단계에서 위험 수준에 따라 단계적으로 대응하는 FDS"로, 티빙식 전면 잠금의 대안을 제시한다.
- **Business Goal:** 부정결제 피해 금액과 차지백 비용을 낮추면서 정상 고객의 결제 이탈을 최소화해 간편결제 신뢰를 유지한다는 가설을 검증한다.

## Success Metrics (SMART, AARRR + Task Success)

- **북극성 지표(North Star Metric):** **월간 부정결제 피해 금액 (차지백 확정 기준, 원)**
  - 이 제품의 핵심 가치(도용당한 돈을 지킨다)를 가장 직접적으로 대표한다. 차단율 같은 중간 지표가 아니라 사용자가 실제로 잃는 금액을 센다.
- **핵심 지표 1 (1단 로그인 탐지력):** RBA 실측 기준 지도학습(GBDT)이 공격 IP를 ROC-AUC 0.86, PR-AUC 0.56으로 탐지. 룰(`ip_user_count ≥ 10`)만으로도 공격의 59%를 정밀도 0.28로 포착(기저 9.5% 대비 약 3배). 출시 후 이 수준 유지를 목표로 한다.
- **핵심 지표 2 (2단 결제 탐지력):** IEEE-CIS 실측 기준 지도학습(LightGBM) AUC 0.826. 기본 기준선에서 Recall 67%, 기준선 조정 시 85~93%. "미탐이 더 치명적"이라는 설계에 따라 **Recall 85% 이상**을 목표로 기준선을 낮춘다.
- **핵심 지표 3 (오탐 통제, 가드레일):** **정상 결제 중 추가 인증, 보류, 차단을 겪은 비율 ≤ 2.0%**. 이 선을 넘으면 피해 금액이 줄어도 정책을 되돌린다. (티빙식 전면 잠금은 이 지표가 100%인 상태)
- **핵심 지표 4 (운영):** 경보 발생부터 최종 조치까지 **P95 조치시간 ≤ 30분**. 상품권 결제는 승인 즉시 사용되므로, 조치가 늦으면 잡아도 소용없다.

> 지표의 연결: 1단(지표1)과 2단(지표2) 탐지력이 오르면 북극성(피해 금액)이 내려간다. 단 오탐(지표3)이 한계선을 넘으면 정상 고객이 이탈하므로, 탐지력을 무한정 밀어붙이지 않고 지표3으로 제어한다.

## Key Scenario

1. **Context →** 김도현의 크리덴셜이 외부 유출로 공격자 손에 넘어갔다.
2. **Problem Moment →** 공격자가 하나의 IP에서 수많은 계정에 로그인을 시도하다가(크리덴셜 스터핑), 김도현 계정에 새벽 3시 낯선 기기와 낯선 국가에서 로그인 성공한다. 이메일을 바꾸고 등록된 카드로 89만 원 상품권을 결제하려 한다.
3. **Product Interaction (2단 방어선) →**
   - **1단(로그인):** RBA 기반 로직이 "같은 IP가 여러 계정을 두드림 + 낯선 국가 + 낯선 기기"를 포착해 이 로그인 세션에 높은 위험 점수를 부여한다.
   - **2단(결제):** 통과한 세션에서 결제가 발생하자, IEEE-CIS 기반 로직이 "평소의 8배 금액 + 상품권 + 이메일 변경 직후"를 결합해 다시 위험 점수를 산출한다.
4. **Resolution →** 두 단계 점수가 모두 고위험이라 결제를 보류하고, 변경 전 원래 연락처(앱 푸시)로 "본인이 맞나요?" 확인을 요청한다.
5. **Outcome →** 김도현은 이 결제를 한 적 없으므로 "아니요"를 누르고, 계정이 보호 잠금된다. 돈이 빠져나가지 않았다.

## Solution & Key Features (MVP)

- **기존 대안의 한계:** 단순 룰(고액 차단, 해외 IP 차단)은 정상 거래를 대량 오탐한다. 티빙식 전면 잠금은 오탐률 100%로 정상 사용자 전원을 막는다. 결제 이력만 보는 FDS는 계정 탈취의 앞 단계(로그인, 이메일 변경)를 놓친다.

- **핵심 기능 (문제와 스토리 연결):**
  1. **2단 방어선 아키텍처** — 1단에서 로그인 단계의 계정 탈취를 거르고, 통과한 세션의 결제를 2단에서 재검증한다. 두 층을 위험 점수로 잇는다. [P0 도용 차단]
  2. **1단 로그인 위험 엔진(RBA 데이터 기반)** — 같은 IP의 다계정 시도, 낯선 국가, 낯선 기기, 로그인 실패 급증을 룰과 지도학습(GBDT)으로 포착. 실측 ROC-AUC 0.86. [P0 계정 탈취 차단]
  3. **2단 결제 위험 엔진(IEEE-CIS 데이터 기반)** — 이메일 도메인 불일치, 금액, 기기, 카드 정보를 지도학습(LightGBM)으로 종합 판단. 실측 AUC 0.826. [P0 부정결제 차단]
  4. **위험 수준별 단계 조치** — 자동 승인, 추가 인증, 결제 보류와 원 연락처 알림, 차단과 계정 잠금. [P0/P1/P2]
  5. **판단 근거 제공(SHAP 등)** — 각 경보에 "왜 위험한지"를 피처 기여도로 설명. 2단 결제 엔진 실측상 카드 나이, 금액, 거래 횟수가 상위 신호. [P1 운영자]

## Scope

### In Scope
- **Must:** 로그인 단계 위험 점수 산출(1단), 결제 단계 위험 점수 산출(2단), 위험 수준별 단계 조치
- **Should:** 두 층을 잇는 위험 점수 파이프라인, 판단 근거(SHAP) 제공, 변경 전 원 연락처 알림
- **Could:** 이상탐지 보조 장치, 그래프 분석 기반 조직적 공격(같은 배송지나 기기로 모이는 계정) 일괄 잠금

### Out of Scope (Won't) + 제외 이유
- **외부 유출 자체의 방지** — 유출은 타 서비스에서 발생하는 보안(정보보호) 사고로, FDS의 범위가 아니다.
- **두 데이터의 계정 키 직접 조인** — RBA(로그인)와 IEEE-CIS(결제)는 서로 다른 서비스라 계정 키로 직접 연결되지 않는다. 공개 데이터의 한계이며, 실제 서비스에서만 가능하다. MVP는 위험 점수를 이어주는 파이프라인으로 시나리오를 완성한다.
- **SIM 스와핑 대응 심화** — 문자 OTP가 뚫리는 SIM 스와핑은 인증 수단 정책(기기 바인딩 등)의 영역으로, MVP에서는 설계 원칙만 명시하고 구현은 제외.
- **실제 페이 내부 데이터 활용** — 결제 비밀번호 변경, 충전 이력 등은 페이사 내부에만 존재해 공개 데이터로 검증 불가. MVP는 공개 데이터로 대체 검증하며 한계를 명시한다.

## UX Principles

- **코어 감정과 Voice:** "지켜주되 방해하지 않는다." 정상 사용자에게는 존재감이 없고, 위험 순간에만 개입한다.
- **정보 우선순위:** 차단이나 인증 화면에서 "왜 멈췄는지"를 먼저, "어떻게 풀지"를 다음에 보여준다.
- **금지와 주의 표현:** "당신이 사기꾼입니다" 식의 단정 금지. "본인 확인이 필요합니다" 식의 중립적이고 보호적인 표현 사용.
- **차단 대신 확인 우선:** 폰 분실이나 해외여행 같은 정상 예외가 있으므로, 즉시 차단이 아니라 "확인 한 번"을 기본값으로 한다. (KCB 교훈)
- **데이터 시각화:** 운영자 화면은 복잡한 차트 대신 "위험도 + 상위 3개 사유(피처 기여도)"를 큰 글씨로 제시.

## Constraints & Assumptions

- **Constraint (고정):**
  - RBA와 IEEE-CIS는 서로 다른 서비스의 데이터라 계정 키로 직접 조인되지 않는다. 두 층은 위험 점수로 잇는다.
  - RBA 원본은 3,300만 건으로 너무 커서, 팀은 앞 50만 행만 잘라서 사용했다. 전체가 아니므로 통계가 편향될 수 있음을 명시한다.
  - RBA는 순수 로그인 데이터라 금액과 결제 정보가 없고, IEEE-CIS는 익명 마스킹(V1~V339)이라 일부 컬럼 의미를 알 수 없다.
  - 결제는 밀리초 단위 판단이 필요하므로, 30일 이력 집계는 사전 계산(리스크 컨텍스트 테이블)으로 미리 준비해야 한다.
  - 라벨(차지백, ATO 판정)은 사후 지연 도착하며, 피해자가 모르면 누락된다.
- **Assumption (검증 필요) + 검증 방법:**
  - "미탐이 오탐보다 치명적이다" → 실제 미탐 1건 평균 피해액(차지백)과 오탐 1건 비용(구매 포기율 × 평균 주문액)을 비교해 검증.
  - "로그인 단계에서 거르면 결제 피해가 줄어든다" → 1단을 통과한 세션과 통과 못한 세션의 결제 사기율을 비교해 검증.
  - "이상탐지는 이 도메인에서 보조에 그친다" → RBA 실측상 IsolationForest는 약했다(공격이 희소 이상치가 아니라 다수 패턴). 결제 층에서도 지도학습 대비 성능을 비교해 재확인.
